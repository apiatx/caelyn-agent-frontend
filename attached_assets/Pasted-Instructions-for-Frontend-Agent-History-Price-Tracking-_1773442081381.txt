Instructions for Frontend Agent — History & Price Tracking Integration

The backend has been updated to use Neon PostgreSQL (accessible in both dev and production). All agent responses are now automatically saved with ticker prices at the time of recommendation and enriched with live prices on retrieval. The frontend needs to call the right endpoints and render the data.

Base URL: your production backend deployment URL (same one the Terminal page already uses for /api/query)

1. History Page — call this endpoint:

GET /api/history
No auth required (or pass existing JWT Bearer token if the user is logged in — the endpoint respects it).

Response shape:

{
  "items": [
    {
      "category": "best_trades",
      "intent": "trades",
      "query": "find me breakout setups today",
      "timestamp": 1741234567.0,
      "display_type": "trades",
      "model_used": "agent_collab",
      "tickers": [
        {
          "ticker": "NVDA",
          "rec_price": 875.00,
          "current_price": 912.50,
          "pct_change": 4.29
        },
        {
          "ticker": "AMD",
          "rec_price": 162.00,
          "current_price": 158.40,
          "pct_change": -2.22
        }
      ],
      "content": "Human-readable summary of the response..."
    }
  ],
  "recent": [ /* same structure, last 10 by default */ ],
  "total_count": 42
}
For the ticker price table on each history item, display: ticker, rec_price (price when agent responded), current_price (live now), pct_change (green if positive, red if negative). rec_price and current_price may be null for non-equity responses (crypto symbols or responses with no price data).

To control how many items come back in recent: GET /api/history?recent_limit=20

2. Terminal Page — History Section:

Call GET /api/history/recent?limit=10 for a lightweight feed of the last N responses. Same shape as above but pre-trimmed — use this instead of the full /api/history to keep the terminal panel fast.

3. Conversation continuity (Terminal Page — already partially wired):

Every response from /api/query returns a conversation_id in the meta field:

{ "meta": { "conversation_id": "ab56be4a-eff2" }, ... }
Store this and pass it back on follow-up queries:

POST /api/query
{ "query": "...", "conversation_id": "ab56be4a-eff2" }
To show a conversation list: GET /api/conversations
To load a full thread: GET /api/conversations/{conv_id}

4. Important notes for the agent:

History is populated automatically — there is no separate "save" call needed. Every completed /api/query response saves to the database on the backend.
pct_change is computed fresh on every /api/history call — live prices are fetched at request time from Finnhub, so the % change is always current.
rec_price is the entry price from the agent's response (e.g. the trade entry price, or current price at time of recommendation). It is null if the agent's response didn't include price data for that ticker.
Only data from after the Neon migration (today) will appear — prior responses were lost because production couldn't connect to the old database. All queries going forward will be stored correctly.
No schema changes needed — the database tables are already created and working. Just call the endpoints.
